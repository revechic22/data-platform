
QBS 작성의 본질
감점 안 당하면서 빨리 처내야 하는 업무입니다. 퀄리티 경쟁이 아니라 기준 충족 + 속도의 문제입니다. 그러면 도구의 핵심 가치는 "판단에 걸리는 시간을 줄여주는 것"입니다. 내용을 직접 만들어주는 게 아니라, "이거 쓸지 말지"를 빠르게 결정할 수 있게 해주는 것.

3가지 방식 비교
방식 A: 현재 방식 (한방 초안 생성)

시간 투자는 개발 80% 완료 상태이고, 실제 사용 시 실행 1분 + HWP 편집 2~3시간입니다. 결과물 완성도는 70~80% 수준의 참고자료이고 HWP 편집은 여전히 수동입니다. 구현 난이도는 현재 거의 완성이고 공정표 범용화만 남았습니다. 실효성은 "뭘 넣어야 하는지" 정리해주는 역할은 하지만, 실제 HWP 작업시간은 크게 줄지 않습니다.

방식 B: 웹 기반 단계별 작성 (제안하신 방식)

시간 투자는 신규 개발 2~4주 예상이고, 실제 사용 시 웹에서 1.1~2.4 순서대로 30분~1시간 + 다운로드 후 HWP 미세조정 30분입니다. 결과물 완성도는 90% 수준이고 각 칸을 직접 확인하고 넣었으니까 다운로드 후 손볼 게 적습니다. 구현 난이도는 중상이고 단계별 UI + 상태관리 + 추천 로직이 핵심입니다. 실효성은 가장 높습니다. "판단 → 입력"이 한 화면에서 끝나고, 빠진 항목이 구조적으로 발생하기 어렵습니다.

방식 C: 하이브리드 (A 기반 + 웹 확인/편집)

시간 투자는 A 완성 후 1~2주 추가이고, 실제 사용 시 자동 생성 → 웹에서 섹션별 확인/수정 → 다운로드입니다. 결과물 완성도는 85~90%입니다. 구현 난이도는 중이고 A의 코드를 Streamlit으로 래핑 + 섹션별 편집 UI 추가입니다. 실효성은 높습니다. A의 자동 생성 속도 + B의 확인/편집 편의성을 결합합니다.

베스트 판단
방식 C(하이브리드)가 가장 현실적입니다.

이유는 세 가지입니다. 첫째, 지금까지 만든 코드를 그대로 활용하므로 개발 시간이 가장 적습니다. 둘째, "한방에 뽑되 웹에서 확인하면서 수정"이라 사용 흐름이 자연스럽습니다. 셋째, 나중에 샘플 DB가 쌓이면 방식 B로 자연스럽게 진화할 수 있습니다.

구체적으로는 이런 흐름입니다:

Copy[1단계] 파일 업로드 (과업지시서 + 평가기준서 + 참고QBS)
    ↓
[2단계] 자동 생성 (현재 generate_draft.py 로직 그대로)
    ↓
[3단계] 웹 화면에서 섹션별 확인
    - 1.1 탭: 기본정보 확인 + 주안점 선택
    - 1.2 탭: 유사사업 5건 확인 + 교체 여부 판단
    - 2.1 탭: 조사항목/분야 대조 확인 + 기간 확인
    - 2.2 탭: 공정표/배점 확인
    - 2.3 탭: 거의 자동 (과업명/발주처만 확인)
    - 2.4 탭: 분야 확인
    ↓
[4단계] 다운로드 (docx)
방식 A에서 방식 C로 전환하는 작업량은 현재 코드 기반으로 Streamlit 래핑 + 탭/섹션별 UI 구성 정도입니다. 현재 남은 작업(공정표 범용화 등)을 마무리한 뒤에 Streamlit 전환을 바로 시작할 수 있습니다.

초기설정
pq팀에서 사업책임기술인이 누군지 그리고 이력을 보내줌
샘플이 수도정비니까 이거 기준으로 설명하면 사책이 신 이야. 예를 들어서, 그럼 과거에 신이름으로 쓴 수도정비 qbs를 몇개 찾아서 저장해둬. 
지자체 로고 및 슬로건을 다운로드, 이거가지고 과업명 까지 참고해서 샘플 hwp 에서 바탕쪽 세팅함


목차
 - 사업수행능력평가기준 및 작성안내서에 나와있는 목차와 완전 동일한지 확인
 - 가장 나중에 수정함

1.1 당해 사업의 내용 및 이해도
 1) 사업의 특성 : 과업지시서 내의 내용을 공간에 맞게 편집해서 넣어줌
 2) 내용 및 범위 : 과업지시서 내의 내용을 가져와서 위치, 기간, 범위 등을 정리하고, 삽도는 지역 및 과업에 맞춰서 과업 지자체시, 군 경계를 기준으로 가운데 놓고, 주변 시, 군, 방위표, 정수장(또는 처리장) 등의 표기 이런식으로 간단하게 만듦.
 3) 과업 수행상 주안점 제시 : 샘플에 있는 내용을 그대로 쓰거나 항목 일부는 다른 샘플로 교체하거나 함. 문구도 그거에 맞게 수정.

1.2 당해 사업 관련 유사사업제시
 - 샘플 자체를 동일 사책의 유사 용역을 가져왔기 때문에 보통 샘플 내용을 그대로 써도 괜찮은 경우가 많음
 - 그러나 pq 팀에서 보내준 자료에 과업별 환산점수가 너무 낮거나 하면 5개중 1,2개는 최대한 점수가 1.0인 과업으로 바꿔줌
    (바꿔야 하는 경우는 또 따로 참고할 자료를 찾아야함 과거 과업 뒤져서 없는 경우도 많아서 웬만하면 샘플에 적힌 내용 그대로 찾거나 다른 샘플에 해당 부분을 보거나 하는 식으로 해결함)
 - 그리고 유사사업 1)~5)까지 입력할때  '금회용역착안 사항' 여기 작성 할때 내용은 경제성 분석 이런식으로 유사한게 들어갈 수 있어도 이미지까지 동일한걸 쓰면 안됨. 성의가 너무 없어 보여. 
 
1.3 사업 성공을 위한 요건 및 추가 제안사항
 - 여기도 거의 샘플 그대로 쓰는데 관련 없는 내용이 들어가있는지는 확인함. 샘플이 정답은 아니기 때문에
 - 1), 3), 4)는 확인만 하는게 일반적이고 그래도 문제가 없는지 보고 수정하긴함. 
 - 2) 설계방법(노선 등)에 따른 제안 및 고려사항은 샘플에서 잘 썼으면 그대로 가고, 이미지가 위 1.2랑 같은거 활용했으면 내용하고 이미지 바꿔줌.
 - 5) 사업효과 극대화 방안 등 도 바로 위에 2) 설명하고 동일한 프로세스로 처리함

 - 당해 사업 관련 유사사업 제시 이건 샘플에꺼를 따라가거나 더 적합한 과업이 있으면 한두개는 수정함.


2.1  당해 사업 관련 업무 수행범위
1) 조사항목 및 내용
 - 과업지시서에서 조사항목 일단 확인하고 샘플도 같이 비교하며서 항목 조정하고 내용 작성
2)  당해 사업 관련 각 분야별 주요 업무
 - 과업에 따라서 분야가 조금씩 차이나는데 사업수행능력평가기준 및 작성안내서 내 조직표 이거보고 분야 확인해서 칸 조정해서 작성함
3) 주요 업무별 특이사항
 - 7페이지 작업계획서랑 연결되는 부분인데 과업지시서에서 일정표가 들어가 있으면 그거를 기준으로 작성함
 - 구분에 들어가는 내용이라던가 특이사항
 - 과업지시서랑 샘플비교해서 작성하고, 구분 내에 보면 기간이 적혀 있는데 이게 과업지시서 상의 일정표에 적힌 기간보다 길게 적으면 안됨.

2.2 작업계획서
 1) 이것도 과업지시서 상의 일정표와 샘플을 비교
 2) 과업지시서에서 개월 수랑 공종을 기본으로 잡고 샘플을 통해서 감하거나 공종을 추가함
 3) 단, 예시로 과업지시서에서 공종 수도정비계획이 1~10개월 잡혀있다면 우리는 그거 이하로 기간을 잡는건 괜찮은데 초과로 잡으면 안됨
 4) 별도로 내가 이것만 만들어 주는 툴을 만들어보려고 간단하게 테스트 해봤음. 관련 내용은 바로 아래
 작업계획서 공정표를 자동으로 만들어주는 웹 앱을 개발하고 있습니다. 최종 목표는 과업지시서 PDF 파일을 업로드하면 AI가 공종, 배점, 기간을 자동으로 읽어내고, 그 값을 종모양 커브로 월별 배분한 뒤 엑셀 공정표로 출력하는 것입니다.

현재까지 Python 환경 세팅, 종모양 배분 알고리즘, 엑셀 출력(대분류/소분류 구조, 색상, 그래프, 누적공정율 포함), 그리고 브라우저에서 접속할 수 있는 Streamlit 웹 앱까지 완성된 상태입니다. 직접 입력 기능도 만들어서 그룹번호와 공종명, 배점, 시작월, 종료월을 CSV 형태로 붙여넣으면 공정표가 생성되는 것까지는 확인했습니다.

다만 가장 핵심인 PDF 자동 추출 기능이 아직 미완성입니다. Google Gemini AI를 연동해서 PDF 텍스트를 읽고 공종/배점/기간을 자동으로 뽑아주는 코드는 작성했지만, API 무료 할당량 문제로 실제 테스트를 못 한 상태입니다. 이 부분이 해결되지 않으면 결국 사람이 값을 직접 입력해야 하기 때문에 기존 작업 방식 대비 이점이 크지 않습니다.


2.3 품질보증체계 및 품질관리계획
 여기부터는 거의 90프로 이상 동일한데 중간에 들어가는 과업명, 발주처명 정도 수정함.
